`stat unit` 是虚幻引擎里排查性能最核心的指令。它展示的数据涵盖了 CPU、GPU 以及渲染管线的关键指标。

# 1. 核心线程耗时

* **Game (游戏线程)**: CPU 处理所有游戏逻辑的时间。包括你的 C++ 算法（例如体素地图的生成与更新）、蓝图逻辑、物理模拟（碰撞检测）、动画更新以及各种 Actor 的 Tick 函数。
* **Draw (绘制线程)**: CPU 准备渲染数据的时间。它不负责画图，而是负责“安排工作”。它会计算当前视野里哪些东西可见（视锥剔除、遮挡剔除），然后把这些可见物体的渲染指令打包成列表。如果你使用大量 ISM/HISM (实例化静态网格体) 构建场景，计算这些实例是否可见的 CPU 开销就会体现在这里。
* **RHIT (RHI 线程)**: CPU 提交渲染指令的时间。RHI 全称是 Render Hardware Interface。Draw 线程打包好虚幻的通用指令后，RHI 线程负责把它们翻译成特定的底层图形 API（如 DirectX 12、Vulkan）能理解的指令，并最终发送给 GPU。
* **GPU time (GPU 时间)**: 显卡真正渲染这一帧画面的绝对时间。包括处理几何体、材质着色、光照计算、硬件光线追踪计算以及后期处理（抗锯齿、泛光等）。

## Frame 与 Game / Draw / GPU 的关系

很多初学者会误以为：`Frame = Game + Draw + GPU`。**这是错误的。**

虚幻引擎采用的是**多线程并行流水线（Pipeline）架构**。这意味着当 GPU 正在渲染第 1 帧画面时，Draw 线程可能正在准备第 2 帧的渲染指令，而 Game 线程已经在计算第 3 帧的逻辑了。它们是同时在不同核心上赛跑的。

因此，它们的关系是著名的**木桶效应（瓶颈原理）**：

**`Frame ≈ MAX(Game, Draw, GPU)`**

* **Frame** 代表的是你最终看到一帧画面所需的总周期耗时（等于 `1000 / 帧率`）。
* 它的数值总是**近似等于 Game、Draw、GPU 这三者中最大的那个数值**（偶尔会因为线程同步等待产生微小误差）。
* **结论**：谁的数值最大，谁就是拖慢整个帧率的“罪魁祸首”。优化数值小的那个线程对提升整体帧数没有任何帮助。

# 2. 渲染与系统统计数据

* **Input**: 引擎处理鼠标、键盘或手柄输入的时间。这个数值通常非常小（小于 1ms），如果不正常飙高，说明输入事件队列堵塞了。
* **Mem**: 当前估算的系统内存（RAM）占用量。
* **RenderRes**: 实际内部渲染分辨率（如 1920x1080）。如果你开启了 TSR、DLSS 或 FSR 等超分辨率技术，这个数值通常会低于你显示器的最终输出分辨率。
* **Draws**: 当前帧的 Draw Call（绘制调用）总数。这是 CPU 命令 GPU 画东西的次数。由于每次调用都有 CPU 开销，这个数字过高会直接压垮 Draw 线程和 RHIT 线程。通过将大量相同材质的方块合并为 ISM/HISM，就是为了极大地降低这个数字。
* **Prims**: 当前帧绘制的图元（Primitives，即三角形/多边形）总数。反映了同屏画面的几何复杂度。
